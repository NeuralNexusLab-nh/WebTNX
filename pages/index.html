WebTNX: Stable & Lightweight HTTP Tunnel

Expose your local development server to the internet. Similar to ngrok, but absolutely free, extremely convenient, and requires no registration. Just use a browser tab or a simple command as your tunnel agent.

What is an HTTP Tunnel?

An HTTP tunnel is a technology that allows you to expose a local server (running behind NATs, firewalls, or without a public IP) to the public internet. It assigns a secure public URL to your local project, allowing external users or APIs (like Webhook senders) to connect directly to your local application from anywhere in the world.

Why can't I just expose my local IP directly?

No Public IP (NAT/CGNAT)

Most home broadband providers use Carrier-Grade NAT. You do not actually have a real public IP address, meaning no one on the internet can locate or reach your computer directly.

Router & Firewall Blockages

Routers block unsolicited inbound traffic. To bypass this, you would have to configure 'Port Forwarding' in your router settings, which is tedious, often restricted, and changes dynamically.

Router Port Limits

ISPs and local router firewalls block incoming traffic on common ports (like 80, 443, 8080) by default to prevent residential web hosting, making direct exposure impossible.

No SSL/HTTPS Encryption

Exposing raw IP usually runs over unencrypted HTTP. Modern web platforms (like Stripe, Line Bot, or browser features like camera access) strictly require HTTPS endpoints to function.

Global Scanning & Bot Probing

Malicious botnets scan raw open ports globally within minutes, attempting automated brute-force attacks and executing exploit payloads directly against your machine.

Lateral Movement & Intranet Infections

If your local computer is compromised through an exposed port, hackers can use it as a pivot to move laterally across your home or corporate local network, infecting other connected devices.

How WebTNX Works

1. Public Request

A visitor accesses your custom WebTNX link (e.g. https://webtnx.zone.id/my-app/).

2. Queue Request

The WebTNX server generates a secure transaction Request ID, logs it into a temporary queue, and holds the visitor's connection open.

3. Agent Polling

Your active WebTNX agent (Web or CLI) polls the server periodically, fetches the pending request, and wipes the server file instantly.

4. Keep-Alive & Local Call

The agent instantly signals the server to sustain the open connection, then forwards the request locally to your localhost.

5. Dynamic Encryption

The agent reads the local response (binary files are safely converted to base64) and encrypts the payload symmetrically using a private security key.

6. Secure Return

The encrypted payload is safely transmitted back. The WebTNX server decrypts it and returns the plaintext content to the visitor.

Local Setup & Requirements

  • Start Local App: Ensure your web application is running locally and listening on a specific port (e.g. http://localhost:8080).
  • Enable CORS (Web Agent Only): If using the Web-based Agent, your local app must accept cross-origin requests from https://webtnx.zone.id.
  • To allow connection, your localhost server must return the following HTTP headers:
    • Access-Control-Allow-Origin: https://webtnx.zone.id
    • Access-Control-Allow-Methods: GET, POST, OPTIONS, PUT, DELETE
    • Access-Control-Allow-Headers: Content-Type, Authorization
  • Keep Connected: If using Web Agent, keep the tab open and prevent sleep mode. If using CLI, keep the terminal running.
