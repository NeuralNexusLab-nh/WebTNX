What is WebTNX? Why WebTNX? Local Host Suitability Troubleshooting Encryption FAQ

1. What is WebTNX?

WebTNX is a stable, convenient, and secure HTTP reverse-proxy tunnel that allows you to safely expose your local servers to the internet without configuring any router settings or system dependencies. Truly capable of stable, long-term server hosting. WebTNX was developed by NeuralNexusLab.

2. Why Build WebTNX Instead of Traditional Tunnels?

Traditional tunneling tools typically require users to download large CLI executables, and set up custom domains, accounts, emails, and API keys. This introduces significant friction in restricted environments or quick debugging scenarios. WebTNX was created to solve these pain points by offering a flexible, account-free architecture that supports both browser-tab agents and lightweight script agents.

3. What is "Local Hosting" & "Proxy Capability"?

A 'Local Host' refers to a live server process running locally on your own computer (e.g. an Express app, a FastAPI service, an Apache server, or Ollama) that is actively listening on a loopback IP (127.0.0.1) and a specific port. For WebTNX to successfully proxy and tunnel your service, your local server must be actively running. If you cannot successfully open your browser, type http://localhost:port, and view your webpage or JSON response locally, WebTNX will not be able to proxy your application.

4. Suitability Matrix: What WebTNX is Best (and Not Best) For

WebTNX is designed as a stable, lightweight, and convenient HTTP tunnel focusing on friction-free local developer testing.

  • Perfect for: Quick Webhook debugging (Line, Stripe, Shopify, GitHub), displaying web prototypes to remote clients, students learning networking, and developers bypassing strict corporate CLI restrictions.
  • Not suitable for: High-throughput media streaming (e.g. hosting local Plex/video servers), heavy continuous high-bandwidth file downloads, production environments requiring 24/7 high availability, and non-HTTP protocols (e.g., SSH, raw TCP/UDP, or MySQL databases).

5. Active Defense & Troubleshooting Guide

  • 502 Connection Refused / Local Service Offline: This means WebTNX cannot find any running process on your specified port. This typically occurs because your local backend application has crashed, your IDE terminal was accidentally closed, or the port number you entered does not match your app's settings. Always test accessing http://localhost:port in your local browser first. If that fails locally, the proxy will also fail.
  • CORS Blockage (Web Agent only): If using the Web-based agent, the browser security sandbox may block requests if CORS headers are missing. To fix this, you must configure CORS headers on your local server. Alternatively, **switch to our Windows Exe, Linux Script, or Python CLI Agent**, which natively bypasses all browser CORS restrictions with zero configuration needed.
  • 504 Gateway Timeout: Your local app took too long to respond. If you are running slow calculations (like local AI token generation on Ollama), please increase the Timeout (Seconds) parameter in the tunnel creation panel.
  • Broken Images / Missing Stylesheets (Relative Paths): WebTNX attempts to rewrite absolute paths (e.g. /css/style.css) to match your custom link. However, if paths are dynamically generated in complex JS, it might fail. For best results, use relative paths (e.g. css/style.css) in your local HTML code.
  • 413 Payload Too Large: If you attempt to transmit payloads or files larger than 2MB, WebTNX server will reject it with 413. This is an intentional security design to safeguard our memory-constrained servers from OOM (Out of Memory) crashes. Compress your assets or limit transfer sizes to resolve this.
  • Polling Connection Warning: If you see polling warning messages in your terminal, it indicates temporary network instability, high server load, or a brief internet drop. The CLI client is designed to automatically retry every 5 seconds. You do not need to restart the application manually.

6. Secure End-to-End Encryption

WebTNX provides basic transit encryption. All request and response data sent between the client and server are encrypted symmetrically to ensure basic data privacy in transit.
